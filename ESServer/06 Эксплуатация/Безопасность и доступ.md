# Безопасность и доступ

## Два контура доступа

Есть два разных контура:

```text
Public API
  доступ desktop-клиента ЭП

Admin UI / Admin API
  доступ администратора
```

Их нельзя смешивать.

## Public API

Для Public API используется API key:

```http
Authorization: ApiKey <keyId>.<secret>
```

Ключ представляет client application. В MVP не делаем жёсткую привязку к рабочему месту, организации или инсталляции.

На стороне desktop-приложения secret хранится в Windows Credential Manager. В конфиге клиента допустимо хранить только `serverUrl` и `keyId`.

## Admin UI

Для Admin UI:

- login/password;
- роль `admin`;
- session cookie;
- CSRF protection для Admin UI/Admin API;
- только внутренний доступ в первом релизе;
- audit всех действий.

## Admin API

Admin API логически выделяется, но физически остаётся внутри `CenteralES.Web`.

Пример:

```text
/admin
/admin/api/*
```

Не создаём отдельный микросервис для Admin API в MVP.

## Внутренний доступ админки

В первом релизе:

- админка не публикуется наружу;
- доступ только во внутренней сети, через bind/reverse proxy/VPN;
- должна быть настройка включения админки;
- должна быть настройка bind URL или allowed networks.

## Audit

В audit пишем:

- вход администратора;
- неуспешные попытки входа;
- создание/отключение/rotation API key;
- изменение endpoint pool;
- изменение pool/endpoint concurrency limits;
- изменение timeout;
- изменение retry policy;
- включение/выключение processor;
- manual retry;
- массовый retry;
- cleanup действия;
- создание/отключение admin user;
- изменение общих настроек.

Минимальные поля audit:

```text
auditId
timestamp
actorAdminId
actorLogin
action
targetType
targetId
oldValue
newValue
reason/comment nullable
correlationId
ip
userAgent
technicalMetadata
```

Комментарий обязателен для:

```text
массовый retry
disable processor
изменение endpoint pool
изменение concurrency limits
изменение retry policy
disable API key
rotate API key
cleanup temporary files вручную
отключение admin user
```

Комментарий не обязателен, но поле можно показать для:

```text
manual retry одной job
создание API key
изменение timeout
health check
просмотр/экспорт support report
```

Audit должен быть append-only. Из админки нельзя редактировать audit-записи и нельзя удалять отдельную запись audit. Raw API key secret, большие payload и входные файлы в audit не пишутся. Старые и новые значения для секретов маскируются.

Текущий backend skeleton:

- хранит audit events в PostgreSQL `admin_audit_events`;
- пишет append-only event `manual_retry_job` при `POST /api/admin/jobs/{jobId}/retry`;
- сохраняет actor admin id/login, action, target, old/new JSON, comment, correlationId, IP/User-Agent и техническую metadata без raw secrets и входных файлов.

## Пароли

Пароли администратора:

- не хранить в открытом виде;
- хранить только hash;
- предпочтительно `Argon2id`;
- допустимый fallback: `bcrypt` или встроенный в .NET `PBKDF2-SHA256`, если `Argon2id` усложнит поставку;
- поддержать смену пароля;
- минимальная длина пароля в MVP: 8 символов;
- seed первого admin-а через init-команду.

Первого администратора создаём не через web setup wizard, а через setup/init command, например:

```text
centerales admin create
```

Команда создаёт первого admin-а только если admin users ещё отсутствуют.

Session baseline:

```text
httpOnly cookie
sameSite=strict/lax
secure=true при HTTPS
idle timeout: 8 hours
absolute timeout: 24 hours
logout invalidates session
```

Текущий backend skeleton:

- хранит admin users в PostgreSQL `admin_users`;
- хранит admin sessions в PostgreSQL `admin_sessions`;
- хранит только hash пароля, session token и CSRF token;
- использует `POST /api/admin/auth/login` для выдачи httpOnly cookie `centerales_admin_session`;
- использует `GET /api/admin/auth/me` для проверки текущей сессии;
- использует `POST /api/admin/auth/logout` для invalidation сессии;
- требует `X-CSRF-Token` для state-changing Admin API endpoint-ов, начиная с logout;
- защищает read-only Admin API endpoint-ы session cookie.

Security events/audit:

```text
admin login success
admin login failed
admin logout
password changed
admin created
admin disabled
```

## API key storage

API key:

- показывается только один раз при создании;
- в БД хранится hash secret-а;
- имеет публичный `keyId`;
- можно отключить;
- можно перевыпустить;
- можно ограничить по capability.
- может иметь `expiresAt`, но в MVP по умолчанию срок действия не задаётся.

При компрометации ключа администратор отключает старый key, создаёт новый, обновляет desktop-клиент и проверяет audit/lastUsed.

Текущий backend skeleton:

- хранит client application в PostgreSQL `client_applications`;
- хранит только PBKDF2-SHA256 hash secret-а;
- проверяет `Authorization: ApiKey <keyId>.<secret>` на Public API;
- возвращает `401 Unauthorized`, если ключ отсутствует, неверен, отключён или истёк;
- возвращает `403 Forbidden`, если ключ валиден, но capability не разрешена;
- обновляет last-used поля только после успешной авторизации.

Admin lifecycle ключей — создание, disable, rotation и audit — реализуется следующим блоком.

## Будущие расширения

Позже можно добавить:

- роли кроме admin;
- HMAC подпись запросов;
- OAuth/JWT;
- IP allowlist;
- MFA для админки;
- обязательный срок действия API keys;
- привязку ключей к организациям, инсталляциям или рабочим местам.
