# Словарь менеджера

Этот словарь объясняет условные термины проекта простым языком. Его можно использовать на встречах с менеджерами, аналитиками и прикладными администраторами.

## Сервер служб ЭП

Серверный компонент для desktop-приложения электронного подписания.

Он выполняет тяжёлые или централизованные операции, которые неудобно или невозможно выполнять на клиентском компьютере.

## Платформа

Основа, на которую можно добавлять разные серверные возможности.

Например, сегодня платформа умеет распознавать PDF, а завтра может обрабатывать изображения или вызывать лицензированный сервис.

## Возможность

Понятная бизнес-функция, которую сервер предоставляет клиенту.

Примеры:

- распознать штамп PDF;
- обработать изображение;
- создать превью документа;
- проверить файл.

Технический термин: `Capability`.

## Обработчик

Конкретный способ выполнить возможность.

Например, возможность “распознать штамп PDF” может выполняться:

- внешним ML-сервисом;
- контейнером;
- лицензированным продуктом;
- нашим кодом.

Технический термин: `Processor`.

## Экземпляр обработчика

Настроенный обработчик в конкретной поставке.

Например, внешний сервис распознавания PDF с пулом endpoint-ов, timeout и лимитами параллельных задач.

Технический термин: `Processor Instance`.

## Пул endpoint-ов

Несколько endpoint-ов одного и того же внешнего сервиса, которые используются как один обработчик для распределения нагрузки.

Например, пять Docker-контейнеров `pdf2txt-http-recognizer` с пятью URL. Для клиента ЭП это всё равно одна возможность распознавания PDF, а для Worker — пул, из которого выбирается свободный и здоровый endpoint.

## Реестр обработчиков

Каталог того, какие возможности и обработчики доступны в системе.

Он помогает серверу понять:

- что включено;
- что выключено;
- каким обработчиком выполнять задачу;
- какие настройки доступны.

Технический термин: `Processor Registry`.

## Задача

Запрос на выполнение обработки.

Например: “обработать этот PDF”.

Технический термин: `Job`.

## Попытка

Одна конкретная попытка выполнить задачу.

Если обработка упала, система может создать новую попытку.

Пример:

```text
попытка 1 - ошибка
попытка 2 - ошибка
попытка 3 - успешно
```

## Hash файла

Уникальный отпечаток содержимого файла.

Если два пользователя отправили один и тот же файл, hash будет одинаковым. Это позволяет не обрабатывать файл повторно.

## Дедупликация

Защита от повторной обработки одного и того же файла.

Если файл уже обрабатывается или результат уже есть, сервер не запускает обработку заново.

## Очередь

Список задач, ожидающих выполнения.

Сервер принимает задачу быстро, кладёт её в очередь, а worker выполняет позже.

## Worker

Фоновая служба, которая выполняет тяжёлую обработку.

Web API принимает запросы, а worker делает долгую работу.

## Web API

Часть сервера, с которой общается desktop-приложение сервиса ЭП.

Она принимает файлы, возвращает jobId, статус и результаты.

## Admin UI

Веб-интерфейс администратора.

Через него можно управлять обработчиками, смотреть очередь, ошибки, ключи доступа и состояние системы.

## Admin API

Внутренний API, через который работает Admin UI.

В первом релизе он не является отдельным микросервисом.

## API key

Ключ доступа для desktop-приложения сервиса ЭП.

Он нужен, чтобы сервер понимал, что запрос пришёл от разрешённого клиента.

## Payload

Полезное содержимое результата.

Это может быть:

- JSON;
- файл;
- набор файлов;
- JSON плюс файлы.

## Artifact

Файл, который является результатом обработки.

Например, обработанное изображение или созданный preview.

## Temporary file

Временный входной файл.

Он нужен только на время обработки и удаляется после завершения.

## Result

Результат обработки.

Он хранится как cache, чтобы при повторном запросе по тому же hash не запускать обработку заново.

## Retry

Повторная попытка обработки после ошибки.

Система может автоматически повторять задачу несколько раз.

## Final failure

Состояние, когда все разрешённые попытки обработки исчерпаны.

После этого обычно требуется действие администратора.

## Health check

Проверка, доступен ли обработчик.

Например, отвечает ли внешний сервис распознавания PDF.

## Runtime-настройки

Настройки, которые можно менять без изменения кода.

Например:

- включить или выключить обработчик;
- поменять endpoint или список endpoint-ов;
- поменять timeout;
- поменять лимит параллельности.

## Поставка

Конкретная установленная версия продукта у заказчика или в среде.

В разных поставках могут быть включены разные обработчики.

## Модульный монолит

Архитектурный подход, при котором система поставляется как единый продукт, но внутри разделена на понятные модули.

Это проще, чем микросервисы, но даёт возможность расширяться.

## Микросервисы

Подход, где части системы являются отдельными сервисами с отдельным deployment и сетевым взаимодействием.

Для первого релиза этот подход считается слишком тяжёлым.
