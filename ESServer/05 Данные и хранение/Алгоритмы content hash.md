# Алгоритмы content hash

Решение от 2026-06-02: content hash для Public PDF upload становится выбираемым алгоритмом, а не жёстко пришитым `SHA-256`.

Дополнение: при upload система считает и сохраняет hash по всем доступным content hash algorithms. Hash выбранного алгоритма остаётся canonical `content_hash` для текущей job/result строки, а остальные значения хранятся как aliases.

Поддерживаемые алгоритмы MVP:

- `sha256` — текущий алгоритм по умолчанию, формат значения `sha256:<hex>`;
- `gost-r-34.11-2012-256` — ГОСТ Р 34.11-2012 / Стрибог-256, формат значения `gost-r-34.11-2012-256:<hex>`.

Выбор алгоритма передаётся параметром upload-запроса. Если параметр не указан, используется `sha256`, чтобы сохранить совместимость существующих клиентов.

Архитектурные правила:

- дедупликация остаётся по `capability + content_hash`;
- префикс алгоритма является частью hash value;
- canonical `content_hash` определяется параметром upload-запроса и сохраняет совместимость текущего job/result контракта;
- все доступные hash-и сохраняются в `processing_content_hashes` как aliases по `subject_id`;
- `GET /api/pdf-stamp-recognition/results/{hash}` принимает prefixed hash любого поддержанного алгоритма и ищет result/job через alias-таблицу;
- временный storage key формируется из безопасного представления полного prefixed hash, без `:` и других символов, опасных для файловой системы;
- security-hash-и для API keys, паролей, session и CSRF не меняются этим решением.

Причина: SHA-256 удобен для общей дедупликации, но для сценариев, связанных с российскими криптостандартами и ЭП, нужен вариант расчёта по ГОСТ Р 34.11-2012. ГОСТ Р 34.11-94 не добавляется, так как он устарел и заменён ГОСТ Р 34.11-2012.
