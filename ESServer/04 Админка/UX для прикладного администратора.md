# UX для прикладного администратора

## Главный принцип

Админка должна говорить языком задач и состояния системы, а не языком внутренних классов.

Плохой пример:

```text
ProcessorInstance heartbeat expired
```

Лучше:

```text
Обработчик PDF не отвечает. Последняя проверка была 12 минут назад.
```

С технической детализацией в раскрываемом блоке.

## Что должен видеть администратор

Администратор должен быстро понимать:

- всё работает или нет;
- какие модули включены;
- какие модули недоступны;
- где очередь растёт;
- какие задачи упали;
- нужна ли ручная реакция;
- хватает ли места;
- действуют ли ключи клиента ЭП.

## Уровни информации

### Уровень 1: Понятный статус

```text
Работает
Отключено
Есть проблемы
Требуется действие
```

### Уровень 2: Причина

```text
Внешний сервис не ответил за 10 минут.
Задача упала 5 раз.
Временное хранилище заполнено на 85%.
API key скоро истечёт.
```

### Уровень 3: Технические детали

```text
HTTP 500
timeout
stack trace
raw error
processor key
job id
correlation id
```

Третий уровень должен быть доступен, но не показываться первым.

## Технический режим

В MVP не нужен отдельный глобальный `DevOps mode`.

Технические данные должны быть доступны через раскрываемые блоки `Технические детали` на конкретных страницах:

```text
Processor Details
Job Details
Result Details
Audit Details
Health Details
```

По умолчанию прикладной администратор видит:

```text
понятный статус
человеческое описание проблемы
влияние на работу
рекомендуемое действие
время последней проверки
переход к нужному экрану
```

Пример:

```text
Обработчик PDF не отвечает.
Новые PDF-задачи временно не обрабатываются.
Последняя успешная проверка: 12 минут назад.
Действие: проверить настройки или выключить обработчик.
```

В раскрываемом блоке `Технические детали` можно показывать:

```text
jobId
correlationId
processor key
capability key
endpoint
endpoint pool
HTTP status
raw error
stack trace
attempt number
heartbeat timestamp
SQL/storage diagnostic summary
```

Главное правило: технические детали должны помогать поддержке и разработчику, но не быть первым уровнем интерфейса для прикладного администратора.

## Быстрые действия

Каждая проблема должна вести к действию:

- обработчик недоступен -> проверить health;
- endpoint или endpoint pool неверный -> открыть настройки;
- job упала -> открыть attempts;
- исчерпаны retry -> manual retry;
- storage заполнен -> открыть storage;
- API key отключён -> открыть ключ.

## Подтверждения

Опасные действия требуют подтверждения:

- выключить обработчик;
- rotate API key;
- удалить временные файлы;
- повторно запустить failed job;
- изменить endpoint или endpoint pool;
- изменить retry policy.

Подтверждение должно объяснять последствия.

## Термины UI

В UI для прикладного администратора лучше использовать:

```text
Возможность
Обработчик
Задача
Попытка
Результат
Ключ доступа
Очередь
Требуется действие
```

Технические термины вроде `Capability`, `Processor`, `Job`, `Payload` можно использовать в документации и техническом режиме, но основной UI должен быть русским и понятным.

## Dashboard

Dashboard должен быть первой рабочей точкой.

Он должен показывать:

- состояние возможностей;
- очередь;
- активные обработки;
- final failures;
- предупреждения storage;
- предупреждения API keys;
- последние задачи.

Выбранный подход:

```text
гибридная сводка + рабочая консоль при проблемах
```

В нормальном состоянии Dashboard показывает общую картину.

В проблемном состоянии он должен перестраивать внимание администратора:

```text
что случилось
почему это важно
какое действие доступно
куда перейти
```

Пример:

```text
2 задачи требуют ручного retry
-> открыть список задач
-> показать фильтр failed_final
-> выбрать retry
-> подтвердить действие
```

## Страница обработчика

Страница обработчика должна совмещать:

- описание;
- статус;
- настройки;
- health;
- последние ошибки;
- связанные jobs.

Это лучше, чем разносить эксплуатацию и конфигурацию по разным местам.

Runtime-параметры должны редактироваться на странице обработчика, но опасные изменения должны проходить через подтверждение.

К опасным изменениям относятся:

```text
endpoint
endpoint pool
retry policy
pool concurrency limit
endpoint concurrency limit
выключение обработчика
```

Подтверждение должно быть написано понятным языком:

```text
Новые задачи PDF будут распределяться по обновлённому endpoint pool.
Текущие задачи продолжат выполняться по уже выбранному обработчику.
После сохранения рекомендуется выполнить проверку работоспособности.
```

Если точное влияние зависит от реализации, UI должен честно показать это как техническую деталь, а не скрывать неопределённость.

## Retry UX

Retry одной задачи выполняется из страницы задачи.

Массовый retry выполняется только из списка задач после фильтрации.

Массовый retry не должен быть одной кнопкой на Dashboard.

Перед массовым retry администратор должен увидеть:

- сколько задач будет перезапущено;
- какой capability затронут;
- какие статусы выбраны;
- есть ли лимит на запуск;
- что действие будет записано в audit.

## Проверки UX

Проверки должны быть разделены по смыслу:

```text
Система
Обработчики
Диагностические тесты
```

Админка может проверять работоспособность, но не должна запускать обычную бизнес-обработку файлов.

Если processor поддерживает безопасный диагностический тест, UI должен явно отделять его от пользовательской обработки:

```text
Проверить работоспособность
```

а не:

```text
Распознать PDF
```

Это снижает риск, что админка станет вторым клиентом системы.

## Состояния без паники

UI должен различать:

- нормальное ожидание;
- временную проблему;
- проблему, требующую администратора.

Например:

```text
processing
  нормальное состояние

retrying
  система сама пытается исправить

failed final
  нужна ручная реакция или изменение политики
```

## Raw JSON и технические данные

Raw JSON нужен, но не должен быть главным экраном.

Для PDF лучше показать:

- найденные люди;
- должности;
- подписанты;
- статус;
- confidence/diagnostics, если есть.

Raw JSON:

- отдельный tab;
- copy/download;
- не мешает основному просмотру.

## Экспорт для поддержки

Экспорт отчёта должен быть отдельным действием, а не автоматической выгрузкой всего подряд.

Для администратора формулировка должна быть понятной:

```text
Скачать отчёт для поддержки
```

а не:

```text
Dump technical context
```

Перед экспортом UI должен объяснить, что попадёт в файл:

- сведения о задаче;
- попытки и ошибки;
- технические идентификаторы;
- краткий health snapshot;
- ссылка на результат.

По умолчанию не включаются большие payload, входные файлы и result artifacts.

Если raw JSON можно экспортировать отдельно, это должно быть отдельное действие.

## Audit UX

В списке audit основным текстом должно быть человеческое описание:

```text
Администратор Иванов изменил endpoint pool обработчика PDF.
```

Технические детали раскрываются отдельно:

```text
oldValue
newValue
correlationId
ip
user agent
metadata
```

Если действие опасное, UI должен дать поле для комментария или причины там, где это действительно полезно.

Комментарий обязателен для:

```text
массовый retry
disable processor
изменение endpoint pool
изменение concurrency limits
изменение retry policy
disable API key
rotate API key
cleanup temporary files вручную
отключение admin user
```

Комментарий не обязателен, но поле можно показать для manual retry одной job, создания API key, изменения timeout, health check и экспорта support report.

Audit должен восприниматься как журнал, а не как редактируемая сущность. В UI нельзя редактировать или удалять отдельные audit-записи.

## Граница MVP

MVP админки должен быть рабочим, но не перегруженным.

В MVP оставляем экраны, которые помогают эксплуатировать первую поставку:

```text
Dashboard
Поставка
Processors
Processor Details
Jobs
Job Details
Results
Result Details
API Keys
Storage
Health
Audit
Admin Users
Settings
```

Откладываем всё, что превращает админку в платформу разработки:

```text
создание processor с нуля
редактор contracts
сложные роли
аналитика
multi-tenant
глобальный DevOps mode
```
