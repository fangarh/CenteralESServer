# Итоговая архитектурная концепция

## Краткое определение

Сервер служб ЭП — это модульная серверная платформа для desktop-приложения сервиса электронного подписания.

Она предоставляет клиенту ЭП набор бизнес-возможностей, а внутри управляет обработчиками, задачами, результатами, файлами, безопасностью и администрированием.

## Архитектурный стиль

Выбран стиль:

```text
Модульный монолит по коду + несколько процессов по эксплуатации
```

Это означает:

- единый solution;
- единая доменная модель платформы;
- единая PostgreSQL;
- чистые границы модулей;
- отдельные entrypoint-процессы для Web и Worker;
- возможность в будущем выделять отдельные части, если это станет оправдано.

## Почему это подходит

Проект должен быть расширяемым, но первый релиз не должен утонуть в инфраструктуре.

Модульный монолит даёт:

- простую разработку;
- контролируемую архитектуру;
- удобное тестирование;
- меньше deployment-сложности;
- возможность расширять систему новыми модулями.

Отдельные процессы дают:

- независимое масштабирование API и обработки;
- устойчивость Web API к долгим задачам;
- возможность перезапускать worker без остановки админки и Public API.

## Главные доменные понятия

```text
Capability
  бизнес-возможность платформы

Processor
  конкретная реализация capability

Processor Instance
  настроенный экземпляр processor в конкретной поставке

Job
  попытка выполнения capability

Result
  результат выполнения capability для конкретного hash
```

## Центральный поток обработки

```text
1. Desktop-клиент ЭП отправляет файл или hash.
2. Web API проверяет API key.
3. Web API определяет capability.
4. Сервер считает content hash.
5. Если результат уже есть, возвращается готовый результат.
6. Если результата нет, создаётся или находится активная job.
7. Worker забирает job из PostgreSQL-очереди.
8. Worker выбирает Processor Instance через Processor Registry.
9. Worker вызывает адаптер обработчика.
10. Результат сохраняется в таблице подсистемы.
11. Общий индекс результата обновляется.
12. Клиент получает результат по hash или отслеживает jobId.
```

## Чистая архитектура

Архитектура должна разделять:

- Domain: правила и сущности;
- Application: use cases;
- Infrastructure: PostgreSQL, файлы, HTTP, внешние сервисы;
- Presentation: Web API, Admin API, Admin UI;
- Worker Host: выполнение фоновых use cases.

Главное правило:

> Domain и Application не должны зависеть от конкретного PostgreSQL, файловой системы, HTTP-клиента или UI.

## DDD-границы

Предварительные bounded contexts:

- Processing Platform: capability, processor, job, result.
- Access Control: API keys, admin users, permissions.
- Admin Operations: настройки, health, audit, ручные действия.
- Storage: temporary files, result payloads, cleanup.
- PDF Stamp Recognition: первый конкретный capability-модуль.

## Что должно быть стабильным

Стабильными должны быть:

- модель capability/job/result;
- контракты Public API;
- правила дедупликации;
- правила retry;
- интерфейсы storage;
- интерфейсы processor adapter;
- общий подход к result payload tables.

## Что может меняться

Могут меняться:

- конкретные обработчики;
- внешний ML endpoint;
- способ хранения файлов;
- способ очереди;
- состав capability;
- вид админки;
- политики retry/retention.

Архитектура должна позволять эти изменения без переписывания ядра.
