# Открытые вопросы

Эта страница фиксирует вопросы, которые ещё нужно согласовать до GSD и реализации.

## Архитектура и deployment

Уже согласовано:

- первый релиз поставляется через Docker Compose;
- Windows Service и Linux daemon не входят в основной MVP-путь;
- основной режим: single-node deployment;
- процессы: Web + один или несколько Worker;
- storage: локальный shared volume между Web и Worker контейнерами;
- PostgreSQL входит в docker-compose поставку или подключается как внешний PostgreSQL по конфигурации;
- несколько worker-процессов на одной машине поддерживаются;
- multi-node с несколькими машинами и общим storage откладывается.
- Web предоставляет `GET /health/live` и `GET /health/ready`;
- Web `/health/live` проверяет только живость HTTP-процесса;
- Web `/health/ready` проверяет конфигурацию, PostgreSQL, совместимость схемы БД, temporary/result storage и hard storage limit;
- Worker не публикуется наружу как HTTP-сервис;
- Worker health для Docker реализуется через локальный healthcheck command или локальный management endpoint внутри контейнера;
- Worker пишет heartbeat каждые `30 seconds`;
- stale heartbeat threshold: `3 minutes`;
- админка показывает статусы `healthy`, `degraded`, `unhealthy`, `unknown`;
- health-check-и не вызывают внешние processor-ы и не запускают обработку файлов без явно безопасного diagnostic contract.

## Админка

Уже согласовано:

- основные действия: упавшие задачи, retry, проверки работоспособности, API keys, поставка компонентов;
- Dashboard гибридный: общая сводка + рабочая консоль при проблемах;
- runtime-параметры компонента управляются на странице компонента;
- опасные изменения требуют подтверждения;
- retry одной задачи и массовый retry по фильтру входят в MVP;
- проверки делятся на Basic health, Processor health и Optional diagnostic test;
- админка не запускает обычную бизнес-обработку файлов.
- отдельный экран “Поставка” входит в MVP и показывает состав текущей установки.
- глобальный DevOps mode в MVP не нужен; технические детали доступны в раскрываемых блоках на конкретных страницах.
- экспорт отчёта для поддержки входит в MVP, но без больших payload по умолчанию.
- audit обязателен для действий администратора.
- audit append-only; комментарий обязателен для опасных действий вроде массового retry, disable processor, изменения endpoint pool/concurrency/retry policy, disable/rotate API key и ручного cleanup.
- `Health` входит в MVP как отдельный экран.
- для внешних сервисов без собственного health endpoint используем `config valid`, `passive health` и ручную диагностику только при явно безопасном сценарии.
- `Results` входит в MVP как поиск и просмотр результата, без аналитики.
- `Admin Users` входит в MVP с единственной ролью `admin`.
- `Settings` остаётся узким общесистемным экраном.
- сложные роли, создание processor-а через UI, редактор contracts, retention editing/cleanup UI, аналитика, пакетный экспорт больших payload, глобальный DevOps mode и multi-tenant откладываются.

## PDF и ML-сервис

- Какой точный JSON-контракт возвращает текущий PDF ML-сервис?
- Можно ли использовать нулевой файл как `InvalidInputProbe`, и какой точный HTTP status/body считать ожидаемым validation error?

Уже согласовано:

- текущий `https://pdf2txt.selectel.dt1520.ru/recognize_json/` считаем чёрным ящиком;
- отдельного health endpoint у сервиса нет, и пока не рассчитываем на его добавление;
- админка не запускает обычную обработку файла ради проверки работоспособности;
- до подтверждения ответа сервиса `InvalidInputProbe` остаётся гипотезой, а не утверждённым контрактом.
- точный JSON-контракт и возможность `InvalidInputProbe` не блокируют старт GSD, а входят в discovery-задачу первой вертикальной фазы;
- диагностика внешнего вызова хранится отдельно от result payload, на уровне attempt;
- successful result хранится как payload подсистемы, diagnostics хранит endpoint, duration, HTTP status, normalized error, retryable и correlationId;
- raw diagnostic excerpt допускается только обрезанным и очищенным, базовый лимит MVP `16 KiB`;
- входные файлы, secrets и большие payload в diagnostics не пишутся.

Классификация ошибок processor-а согласована:

```text
invalid_input
processor_timeout
processor_unreachable
processor_http_error
processor_bad_response
processor_contract_error
processor_overloaded
temporary_storage_full
internal_error
```

`processor_overloaded` не создаёт failed attempt. Job остаётся в очереди или получает короткий delay.

## API

Уже согласовано:

- `POST /api/pdf-stamp-recognition/jobs`, `GET /api/jobs/{jobId}`, `GET /api/pdf-stamp-recognition/results/{hash}` остаются baseline endpoint-ами;
- если result ещё не готов, `GET /results/{hash}` возвращает `202 Accepted`;
- если result отсутствует и активной обработки нет, `GET /results/{hash}` возвращает `404 Not Found`;
- отдельный status endpoint по hash в MVP не добавляем;
- если клиенту нужны детали конкретной попытки, он использует `GET /api/jobs/{jobId}`;
- для `POST /jobs` content hash достаточен как идемпотентность первого релиза;
- endpoint отмены job не входит в MVP;
- при исчерпании retry наружу возвращаем статус `blocked`;
- Public API использует единый формат ошибки с `correlationId`;
- raw errors внешних processor-ов клиенту ЭП не отдаём.

## Лимиты и timeout-ы

Уже согласовано:

- default max upload size: `250 MiB`;
- hard safety max upload size: `500 MiB`;
- при превышении лимита файла Public API возвращает `413 Payload Too Large`;
- connect timeout внешнего processor-а: `15 seconds`;
- overall processor timeout: `15 minutes`;
- max recommended processor timeout: `60 minutes`;
- default `endpointConcurrencyLimit`: `2`;
- default `poolConcurrencyLimit`: `endpoint count * endpointConcurrencyLimit`;
- если все endpoint-ы заняты, job не падает, а остаётся в очереди или получает короткий delay `10-30 seconds`;
- heartbeat interval: `30 seconds`;
- stale heartbeat threshold: `3 minutes`.

## Безопасность

Уже согласовано:

- Public API использует `Authorization: ApiKey <keyId>.<secret>`;
- на сервере хранится только hash secret-а;
- на стороне desktop-приложения secret хранится в Windows Credential Manager;
- в конфиге desktop-приложения можно хранить только `serverUrl` и `keyId`;
- `expiresAt` поддерживается моделью, но по умолчанию не задан;
- жёсткая привязка API key к рабочему месту, организации или инсталляции не входит в MVP;
- ротация ключа в MVP ручная;
- Admin UI использует login/password, одну роль `admin`, session cookie и CSRF protection;
- минимальная длина пароля администратора: 8 символов;
- пароль хранится только hash-ом, предпочтительно `Argon2id`, fallback `bcrypt`;
- первый admin создаётся через init command, web setup wizard не входит в MVP.

## Хранение

Уже согласовано:

- temporary storage soft limit: `80%`;
- temporary storage hard limit: `90%`;
- minimum free space: `10 GiB`;
- при hard limit Web перестаёт принимать новые файлы;
- `POST /jobs` возвращает `503 Service Unavailable` с кодом `temporary_storage_full`;
- temporary input после `completed` удаляется Worker lifecycle cleanup-ом;
- temporary input после `failed`/`blocked` сохраняется для manual retry;
- orphan/abandoned файлы пока не удаляются автоматически: будущий безопасный путь — dry-run кандидатов после grace period `24 hours`, затем отдельное audited cleanup action;
- отдельное приложенческое сжатие JSON-результатов в MVP не вводим;
- result index хранит `payload_size`;
- большие payload не читаются в списках и search endpoints;
- модель допускает будущий перенос крупных JSON в compressed artifact storage;
- результаты в MVP хранятся бессрочно как cache;
- read-only retention policy visibility уже есть в Admin Storage/Settings;
- активное удаление результатов, cleanup worker и ручная cleanup-кнопка откладываются до отдельного checkpoint-а с confirmation, audit и проверкой влияния на manual retry.

## Процесс

Уже согласовано:

- git-репозиторий создан: `https://github.com/fangarh/CenteralESServer`;
- `.superpowers/` не хранится в репозитории;
- Obsidian-заметки в `ESServer` являются источником истины до старта GSD.
- после старта GSD Obsidian остаётся архитектурным источником истины;
- GSD-артефакты считаются производными рабочими артефактами планирования и исполнения;
- если GSD-план конфликтует с Obsidian-решением, сначала обновляется Obsidian-решение, затем GSD-план;
- первыми GSD-артефактами создаём project/milestone контекст, roadmap MVP и план первой вертикальной фазы;
- первый MVP milestone строим вокруг рабочего vertical slice: Public API -> PostgreSQL queue -> Worker -> pdf2txt adapter -> result retrieval -> минимальная админская видимость.
- discovery контракта `pdf2txt` входит в первую фазу и выполняется до финализации adapter contract.
