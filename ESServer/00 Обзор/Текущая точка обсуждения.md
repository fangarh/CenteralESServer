## Что уже зафиксировано

- Строим не один PDF endpoint, а серверную платформу для desktop-приложения сервиса ЭП.
- Первый практический сценарий: распознавание штампа PDF и получение авторов, должностей и подписантов.
- Внешний PDF ML-сервис считается чёрным ящиком: его внутреннюю логику и JSON-контракт результата не меняем.
- Архитектурная форма: модульный монолит по коду, несколько процессов по эксплуатации.
- Минимальные процессы: `CenteralES.Web` и `CenteralES.Worker`.
- `CenteralES.Web` содержит Public API, Admin API, Admin UI hosting и auth.
- `CenteralES.Worker` выполняет тяжёлую обработку, retry, вызов processor-ов и cleanup.
- Admin API выделяется логически, но не как отдельный микросервис.
- Внешний брокер очередей в MVP не добавляем: используем PostgreSQL-backed queue.
- Для конкурентного выбора задач worker-ами ориентируемся на `FOR UPDATE SKIP LOCKED`.

## Модель возможностей и обработчиков

Внутренняя модель:

```text
Capability -> Processor -> Processor Instance -> Job -> Result
```

Простое объяснение:

- `Capability` — бизнес-возможность, которую просит клиент ЭП.
- `Processor` — конкретный способ выполнить эту возможность.
- `Processor Instance` — настроенный экземпляр обработчика в конкретной поставке.
- `Job` — конкретная попытка выполнения.
- `Result` — результат обработки по capability и hash.

В первом релизе администратор не создаёт новые обработчики с нуля. Разработчик заранее описывает их в коде или конфигурации поставки, а администратор управляет только тем, что уже включено.

Для конфигурации выбран гибрид:

```text
Definition в коде
Runtime-настройки в БД и админке
```

В коде задаются capability key, processor key, тип, adapter, result store, контракты, default limits и schema разрешённых настроек.

В БД и админке хранятся enabled/disabled, endpoint, timeout, retry policy, concurrency limit, credentials/license refs, health status и last error.

## Public API

Клиент ЭП должен работать с понятными доменными endpoint-ами, а не с абстрактным `/capabilities/{key}` как основным интерфейсом.

Пример для PDF:

```http
POST /api/pdf-stamp-recognition/jobs
GET  /api/pdf-stamp-recognition/results/{hash}
GET  /api/jobs/{jobId}
```

Если один и тот же файл отправили из двух мест:

- повторная обработка не запускается;
- система ищет активную обработку по `capability + content_hash`;
- клиентам возвращается общий текущий `jobId`;
- разные части клиента могут отслеживать один и тот же результат.

## Retry и job

`jobId` считается идентификатором конкретной попытки обработки.

Если обработка упала, система может создать новую попытку:

```text
attempt 1 -> failed
attempt 2 -> failed
attempt 3 -> processing
```

Клиенту нужно возвращать:

- hash;
- текущий или новый `jobId`;
- номер попытки;
- статус;
- информацию, что система повторяет обработку.

После достижения лимита повторов поведение должно настраиваться в Admin UI. Безопасный режим по умолчанию: ручной retry через админку.

## Хранение

Входные файлы:

- временные;
- не хранятся для истории;
- в MVP хранятся локально;
- удаляются после завершения обработки или final failure;
- архитектурно должны идти через `TemporaryFileStorage`, чтобы позже перейти на S3/MinIO.

Результаты:

- хранятся как cache по `capability + content_hash`;
- в MVP хранятся бессрочно;
- общая таблица результатов хранит только индекс и ссылку;
- payload хранится в таблицах конкретных подсистем;
- результат может быть не только JSON, но и файл, набор файлов или JSON плюс файлы.

Retention результатов в MVP не реализуется, но модель должна предусматривать future policy.

## Авторизация и доступ

Для Public API выбран API key A+:

```http
Authorization: ApiKey <secret>
```

Ключи должны:

- храниться только hash-ом;
- выпускаться и отключаться через админку;
- иметь allowed capabilities;
- иметь audit и last used.

Для админки в первом релизе:

- без внешних публичных endpoint-ов;
- login/password;
- только роль `admin`;
- модель должна позволять роли в будущем.

## Админка

Админка входит в MVP и должна быть качественной частью продукта.

Выбран формат:

```text
гибрид эксплуатационной панели и конфигурационной панели поставки
```

Пользователь админки:

```text
прикладной администратор, не DevOps
```

Это значит:

- UI говорит языком задач и состояния системы;
- технические детали доступны, но не показываются первыми;
- каждая проблема должна вести к понятному действию;
- Dashboard должен быть рабочей консолью, а не декоративной страницей.

Предварительная структура MVP:

```text
Dashboard
Поставка
Processors
Processor Details
Jobs
Job Details
Results
Result Details
API Keys
Storage
Health
Audit
Settings
Admin Users
```

Согласованные сценарии MVP:

```text
упавшие задачи
запуск retry
проверки работоспособности
управление API keys
поставка компонентов
```

Dashboard выбран как гибрид:

- в нормальном состоянии показывает состояние системы, возможности, очередь, ключи и storage;
- при проблемах становится рабочей консолью и ведёт к конкретному действию.

Runtime-параметры компонента управляются на странице компонента.

Опасные изменения требуют подтверждения:

```text
endpoint
retry policy
concurrency limit
выключение компонента
```

Retry:

- retry одной задачи из `Job Details`;
- массовый retry по фильтру из списка задач;
- массовый retry только после предпросмотра, подтверждения и записи в audit.

Проверки работоспособности:

```text
Basic health
Processor health
Optional diagnostic test
```

Админка не запускает обычную бизнес-обработку файлов. Она может выполнять только проверки работоспособности и безопасные диагностические тесты, если они явно поддержаны processor definition.

Отдельный экран `Поставка` входит в MVP.

Он нужен как карта состава текущей установки:

- какие capability входят в поставку;
- какие capability недоступны;
- какие processor-ы доступны;
- что включено runtime;
- что требует настройки или внимания.

Технический режим:

- глобальный `DevOps mode` в MVP не нужен;
- по умолчанию UI показывает понятный статус, влияние на работу и рекомендуемое действие;
- технические поля раскрываются в блоках `Технические детали` на конкретных страницах.

В технических деталях допустимы:

```text
jobId
correlationId
processor key
capability key
endpoint
HTTP status
raw error
stack trace
attempt number
heartbeat timestamp
SQL/storage diagnostic summary
```

Экспорт отчёта для поддержки входит в MVP.

Минимальный экспорт доступен из `Job Details`; если результат уже есть, `Result Details` может давать связанный экспорт контекста результата.

По умолчанию отчёт не включает входной файл, большие payload и большие result artifacts.

Audit обязателен для действий администратора.

Минимальные поля audit:

```text
кто
когда
действие
объект
старое значение
новое значение
причина/комментарий, если нужен
correlationId
ip/user agent, если применимо
technical metadata
```

`Health` входит в MVP как отдельный экран. Dashboard показывает summary, а `Health` показывает подробное состояние Web, Worker, PostgreSQL, storage, processor health, очереди и heartbeat.

`Results` входит в MVP как поиск и просмотр результата, без аналитики.

`Admin Users` входит в MVP с единственной ролью `admin`.

`Settings` остаётся узким общесистемным экраном.

Из MVP откладываются:

```text
сложные роли кроме admin
создание processor с нуля через UI
редактор произвольных contracts
retention policy UI для результатов
полноценная аналитика и графики
экспорт больших payload пакетами
глобальный DevOps mode
multi-tenant и организации
```
