# Краткое резюме договорённостей

## Что строим

Строим серверную платформу для desktop-приложения сервиса электронного подписания. Платформа предоставляет набор серверных возможностей, которые можно включать, отключать и настраивать в поставке.

Это не просто один REST endpoint для PDF. Это основа для будущего набора серверных плагинов: PDF, изображения, лицензированные обработчики, внутренние алгоритмы и будущие подсистемы.

## Технологические рамки

- .NET 9.
- PostgreSQL.
- Entity Framework не используем.
- Чистая архитектура.
- DDD.
- TDD.
- Внешние ML/лицензированные сервисы считаются чёрными ящиками.

## Архитектурная форма

Выбран подход:

> Модульный монолит по коду, несколько процессов по эксплуатации.

Это означает:

- код организован как единая система с чистыми границами модулей;
- физически в поставке есть минимум два процесса: Web и Worker;
- Web отвечает за API, админку и постановку задач;
- Worker отвечает за тяжёлую обработку, retry, вызов обработчиков и cleanup;
- это не тяжёлые микросервисы с отдельными базами и сетевыми контрактами между каждым модулем.

## Обработчики

Главный термин: `Capability`.

`Capability` означает бизнес-возможность платформы, например:

- распознать штамп PDF;
- обработать изображение;
- сделать превью документа;
- проверить файл;
- выполнить будущую серверную функцию для сервиса ЭП.

`Processor` означает конкретную реализацию этой возможности:

- внешний REST-сервис;
- контейнер;
- лицензированный продукт;
- наш внутренний код.

В MVP обработчики не создаются администратором с нуля. Разработчик заранее описывает их в коде или конфигурации поставки. Администратор управляет только тем, что уже включено в поставку.

Для внешних HTTP processor-ов один processor instance может содержать endpoint pool. Например, пять Docker-контейнеров `pdf2txt-http-recognizer` с пятью URL считаются одним processor instance и используются для распределения нагрузки.

Для endpoint pool согласовано:

- общий `poolConcurrencyLimit`;
- `endpointConcurrencyLimit` для каждого endpoint-а;
- default `endpointConcurrencyLimit`: `2`;
- default `poolConcurrencyLimit`: количество endpoint-ов * `2`;
- выбор endpoint-а на момент выполнения attempt;
- алгоритм MVP: `least in-flight`;
- выбранный endpoint сохраняется в истории attempt;
- если endpoint падает, падает только текущая attempt, а retry может уйти на другой endpoint.

Лимиты PDF и внешнего processor-а:

- default max upload size: `250 MiB`;
- hard safety max upload size: `500 MiB`;
- connect timeout: `15 seconds`;
- overall processor timeout: `15 minutes`;
- max recommended processor timeout: `60 minutes`;
- если все endpoint-ы заняты, job не падает, а остаётся в очереди или получает delay `10-30 seconds`.

## PDF сценарий

Первый сценарий: распознавание штампа PDF и получение авторов, должностей и подписантов.

Текущий ML-сервис:

```text
https://pdf2txt.selectel.dt1520.ru/recognize_json/
```

Он считается чёрным ящиком. Мы не меняем его внутреннюю логику и не меняем контракт результата.

Отдельного health endpoint у текущего сервиса нет. В MVP состояние такого внешнего сервиса показываем осторожно: локальная валидность настроек, passive health по последним реальным вызовам и ручная диагностика только при явно безопасном сценарии. Проверка нулевым файлом возможна как будущий `InvalidInputProbe`, но будет утверждена только после проверки фактического ответа сервиса.

Сценарий должен быть асинхронным:

1. Клиент отправляет файл.
2. Сервер считает hash.
3. Если результат уже есть, сервер возвращает готовый результат.
4. Если результата нет, сервер создаёт job.
5. Worker обрабатывает job.
6. Клиент получает статус и результат по hash/job.

## Дедупликация

Если один и тот же файл отправили из двух мест, повторная обработка не запускается.

Для пары:

```text
capability + content_hash
```

должна быть одна актуальная активная обработка.

Клиентам возвращается один и тот же текущий `jobId`, чтобы разные части системы могли отслеживать общий результат.

## Retry

`jobId` считается идентификатором конкретной попытки обработки.

Если обработка упала, система может автоматически создать новую попытку:

```text
attempt 1 -> failed
attempt 2 -> failed
attempt 3 -> processing
```

После достижения лимита повторов поведение настраивается в админке:

- заблокировать до ручного retry администратора;
- разрешить клиенту создать новую серию попыток;
- разрешить после cooldown.

Безопасный режим по умолчанию: ручной retry через админку.

## Хранение файлов

Входные файлы хранятся временно.

Для MVP выбран локальный storage:

- проще поставка;
- не нужен S3/MinIO на первом релизе;
- подходит, если файл нужен только до окончания обработки.

Архитектурно должна быть абстракция `TemporaryFileStorage`, чтобы позже можно было перейти на S3-compatible storage.

Temporary storage limits:

- soft limit: `80%`;
- hard limit: `90%`;
- minimum free space: `10 GiB`;
- при hard limit новые upload-ы не принимаются, Public API возвращает `503 temporary_storage_full`;
- temporary input files удаляются после завершения обработки, блокировки, отмены или после grace period для abandoned.

## Хранение результатов

Результаты хранятся как cache по `capability + hash`.

Общая таблица результатов не должна хранить большие payload.

Общее правило:

- общая таблица хранит индекс и ссылку;
- payload хранится в таблицах конкретных подсистем;
- результат может быть JSON, файл, набор файлов или JSON плюс файлы.

Для PDF в MVP результатом является существующий JSON-контракт.

## Очередь задач

Для MVP выбрана PostgreSQL-backed queue.

Внешний брокер RabbitMQ/Redis/Kafka/NATS пока не добавляем.

Worker-ы забирают задачи из PostgreSQL через безопасный конкурентный механизм, например `FOR UPDATE SKIP LOCKED`.

## Deployment MVP

Для первого релиза выбран:

```text
Docker Compose
single-node deployment
Web + один или несколько Worker
PostgreSQL
локальный shared storage volume
```

Docker считается основным способом поставки и снимает необходимость делать Windows Service в MVP.

Windows Service и Linux daemon не входят в основной путь первого релиза.

Multi-node deployment с несколькими машинами и общим storage откладывается.

## API

Выбран гибридный API:

- снаружи клиент ЭП работает с понятными доменными endpoint-ами;
- внутри всё реализуется через общую модель `Capability -> Job -> Processor -> Result`.

Пример:

```http
POST /api/pdf-stamp-recognition/jobs
GET  /api/pdf-stamp-recognition/results/{hash}
GET  /api/jobs/{jobId}
```

Для Public API согласован baseline:

- `POST /jobs`: `200 OK`, если result уже есть;
- `POST /jobs`: `202 Accepted`, если создана новая job или уже есть активная job по тому же hash;
- `GET /results/{hash}`: `200 OK`, если result есть;
- `GET /results/{hash}`: `202 Accepted`, если обработка ещё идёт;
- `GET /results/{hash}`: `404 Not Found`, если result нет и активной обработки нет;
- при исчерпании retry наружу возвращается статус `blocked`;
- endpoint отмены job не входит в MVP;
- ошибки Public API возвращаются в едином формате с `correlationId`;
- raw errors внешних processor-ов клиенту ЭП не отдаём.

Для внешних processor-ов согласованы нормализованные ошибки:

```text
invalid_input
processor_timeout
processor_unreachable
processor_http_error
processor_bad_response
processor_contract_error
processor_overloaded
temporary_storage_full
internal_error
```

`processor_overloaded` не создаёт failed attempt: job остаётся в очереди или получает короткий delay.

## Авторизация

Для Public API выбран API key A+:

```http
Authorization: ApiKey <keyId>.<secret>
```

Ключи:

- на сервере хранятся только в виде hash secret-а;
- имеют публичный `keyId`;
- выпускаются и отключаются через админку;
- могут быть ограничены по allowed capabilities;
- имеют audit и last used;
- на стороне desktop-приложения secret хранится в Windows Credential Manager;
- `expiresAt` поддерживается моделью, но по умолчанию не задан;
- жёсткая привязка к рабочему месту, организации или инсталляции не входит в MVP.

Ротация ключа в MVP ручная: создать новый key, прописать его в desktop-приложении, проверить работу, отключить старый key.

Audit обязателен для admin actions: retry, массовый retry, изменения processor settings, endpoint pool, concurrency limits, timeout, retry policy, API keys, admin users, cleanup и settings.

Комментарий обязателен для опасных действий: массовый retry, disable processor, изменение endpoint pool, изменение concurrency limits, изменение retry policy, disable/rotate API key, ручной cleanup temporary files и отключение admin user.

Audit append-only. Raw secrets, входные файлы и большие payload в audit не пишутся.

Для админки:

- login/password;
- только роль `admin` в первом релизе;
- session cookie;
- CSRF protection для Admin UI/Admin API;
- модель должна позволять роли в будущем.

Пароль администратора: минимум 8 символов, хранится только hash-ом, предпочтительно `Argon2id`, допустимый fallback `bcrypt`. Первый admin создаётся через init command, web setup wizard в MVP не делаем.

## Админка

Админка входит в MVP.

Пользователь админки: прикладной администратор, не DevOps.

Значит UI должен объяснять состояние системы понятными словами, показывать проблемы и предлагать действия, а не заставлять читать технические логи.

Админка должна быть гибридом:

- эксплуатационная панель;
- конфигурационная панель поставки.

В первом релизе админка не публикуется наружу как внешний endpoint.
