## Текущая архитектура

CenteralESServer остаётся модульным .NET 9 backend MVP:

```text
CenteralES.Web
CenteralES.Worker
CenteralES.DatabaseMigrator
CenteralES.Admin.Bootstrap.WinForms
PostgreSQL
local temporary storage
external pdf2txt HTTP recognizer
```

Ключевые решения:

- PostgreSQL-backed queue без Entity Framework.
- Public API и Admin API разделены.
- Public API защищён API key.
- Admin API защищён admin session cookie; state-changing endpoints требуют CSRF.
- Worker выполняет асинхронную обработку PDF, retry, heartbeat и cleanup.
- SQL migrations применяются явным runner-ом без EF.
- Production path имеет отдельный `CenteralES.DatabaseMigrator`; Web/Worker в Production не должны требовать DDL/bootstrap прав.
- Docker Compose остаётся целевым MVP delivery path.

## Code Review Remediation

Закрытые архитектурные исправления:

- Queue transitions `Complete`, `Defer`, `Fail` теперь требуют `status = processing`, правильный `subject_id` и текущий `processing_subjects.current_job_id`.
- `ClaimNext` переводит и job, и subject в `processing`.
- Public PDF upload orchestration вынесен из endpoint-а в module use-case handler `SubmitPdfStampRecognitionJobHandler`.
- `IProcessingJobQueue` разделён на роли:
  - `IProcessingJobCommandQueue`
  - `IProcessingJobClaimQueue`
  - `IProcessingJobRecoveryQueue`
  - `IProcessingJobReadStore`
- Admin read surface разделён на роли и concrete PostgreSQL stores:
  - `PostgresAdminJobReadStore`
  - `PostgresAdminProcessorReadStore`
  - `PostgresAdminAuditReadStore`
  - `PostgresAdminResultReadStore`
- Старый `PostgresAdminProcessingReadStore` оставлен только composite facade.
- Admin Job Details больше не отдаёт `temporaryFileKey` и raw diagnostics наружу.
- Support report не включает входной PDF, temporary storage key и raw result payload.
- `PostgresPdfStampRecognitionResultStore` заменяет старый JSON payload для того же hash и не оставляет orphan rows.
- Public PDF upload поддерживает выбираемый content hash algorithm: `sha256` по умолчанию и `gost-r-34.11-2012-256` / Стрибог-256 через параметр `hashAlgorithm`; выбранный алгоритм остаётся canonical `content_hash`, а все поддержанные hash-и сохраняются в `processing_content_hashes` как aliases.
- `NpgsqlDataSource` регистрируется как singleton factory.
- `HttpClient` оставлен DI-owned singleton; `IHttpClientFactory` не подключался, потому что пакет `Microsoft.Extensions.Http` не добавлялся.
- Production `appsettings.Production.json` добавлен для Web и Worker; `Database:AutoBootstrap=false`.

## Worker Recovery

Добавлен recovery зависших processing jobs.

Поведение:

- Worker периодически вызывает `IProcessingJobRecoveryQueue`.
- PostgreSQL SQL выбирает stale `processing` jobs через `FOR UPDATE SKIP LOCKED`.
- Восстанавливается только job, которая всё ещё является `processing_subjects.current_job_id`.
- Job возвращается в `queued`.
- `attempt_number` не меняется.
- Новая attempt не создаётся.
- Failed diagnostics не пишутся.
- `started_at` и `heartbeat_at` очищаются.
- `scheduled_at` выставляется в момент recovery.

Worker recovery defaults:

```text
enabled: true
staleJobTimeout: 00:05:00
recoveryInterval: 00:01:00
batchSize: 50
```

Конфигурация:

```json
{
  "PdfStampRecognition": {
    "Recovery": {
      "enabled": true,
      "staleJobTimeout": "00:05:00",
      "recoveryInterval": "00:01:00",
      "batchSize": 50
    }
  }
}
```

## Public API

```http
POST /api/pdf-stamp-recognition/jobs
GET  /api/pdf-stamp-recognition/results/{hash}
GET  /api/jobs/{jobId}
```

Поведение:

- upload size guard;
- temporary storage capacity guard;
- status/result API не отдаёт raw processor errors и storage keys.

## Ограничения

- Не использовать Entity Framework.
- Public API и Admin API не смешивать.
- Health/admin diagnostics не должны запускать обычную business processing через внешний processor.
- Не выводить и не сохранять secrets.
- Raw payload и входные PDF не показывать в обычной админке/support report.
- DB-backed integration tests лучше гонять последовательно из-за общей PostgreSQL test database.
