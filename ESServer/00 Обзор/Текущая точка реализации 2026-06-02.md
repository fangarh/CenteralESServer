## Текущая архитектура

Текущий MVP остается модульным монолитом по коду и несколькими процессами в эксплуатации:

```text
CenteralES.Web
CenteralES.Worker
PostgreSQL
local temporary storage
external pdf2txt HTTP recognizer
```

Основные решения:

- .NET 9.
- PostgreSQL-backed queue без Entity Framework.
- Public API защищен API key.
- Admin API защищен admin session cookie, state-changing endpoints требуют CSRF.
- Worker забирает задачи через PostgreSQL queue, пишет heartbeat и выполняет retry/cleanup.
- PostgreSQL schema bootstrap выполняется explicit SQL migration runner-ом без EF.
- Docker Compose выбран как целевой путь поставки MVP; Windows Service не нужен.
- Дома Docker нет, поэтому Docker-поставку пока не трогали.

## Public API и processing baseline

Реализовано:

- `POST /api/pdf-stamp-recognition/jobs`.
- `GET /api/pdf-stamp-recognition/results/{hash}`.
- `GET /api/jobs/{jobId}`.
- дедупликация по `capability + content_hash`;
- временное хранение входных PDF;
- result index и payload store для результатов;
- retry policy с `maxAttempts`;
- terminal `blocked` после исчерпания попыток;
- normalized processor errors;
- upload size guard;
- temporary storage hard/min-free guard;
- `/health/live` и `/health/ready`.
- file-backed SQL migrations через `PostgresDatabaseBootstrapper.ApplySchemaAsync`.

Для `pdf2txt-http-recognizer`:

- endpoint pool считается одним processor instance;
- распределение нагрузки делается на уровне endpoint pool;
- выбран MVP-алгоритм `least in-flight`;
- endpoint сохраняется в diagnostics attempt-а;
- admin health не вызывает внешний сервис как обычную бизнес-обработку.

## Admin Settings checkpoint

`GET /api/admin/settings` добавлен как read-only endpoint.

Показывает безопасную runtime-сводку:

- Public API max upload bytes;
- temporary storage root/limits;
- processor key/capability;
- recognizer mode;
- sanitized endpoint pool;
- endpoint count;
- pool/endpoint concurrency limits;
- timeout;
- max attempts;
- processor overloaded delay;
- contract version;
- MVP boundary: read-only, editing disabled.

Не возвращает:

- credentials;
- connection strings;
- env var names;
- passwords;
- hashes;
- raw secret values.

Редактирование settings остается отдельным будущим checkpoint-ом с audit, подтверждением и валидацией влияния на новые задачи.

## Admin Audit checkpoint

`ADMIN-03` закрыт для текущего MVP-набора опасных действий.

Сейчас audit пишется для:

- manual retry job;
- create/disable API key;
- create/disable admin user;
- change admin password.

`/admin -> Аудит` теперь умеет:

- фильтровать по action;
- фильтровать по target type;
- фильтровать по target id;
- фильтровать по actor;
- фильтровать по диапазону дат;
- задавать limit;
- показывать количество событий;
- показывать summary примененного фильтра;
- раскрывать безопасные детали события.

Безопасные детали включают:

- audit id;
- action;
- target type/id;
- actor login/id;
- occurred at;
- correlationId;
- comment.

Raw value JSON, credentials, hashes, входные PDF и большие payload не выводятся.

## Что дальше

Так как Docker исключен из текущего запроса, следующий логичный шаг без Docker:

```text
Init/admin bootstrap command
```

Зачем:

- первый admin сейчас нужен для входа;
- `logon.env` есть локально, но нужен нормальный повторяемый bootstrap;
- для MVP-поставки нужен понятный путь создать первого администратора без web setup wizard;
- это пригодится перед Docker Compose.

Предлагаемый checkpoint:

- CLI или отдельный command path для создания первого admin;
- чтение логина/пароля из env/аргументов без записи секретов в Git;
- idempotent поведение;
- audit/логирование без пароля;
- документация локального запуска.

После этого логичный следующий крупный блок:

```text
Docker Compose Delivery MVP
```

Он включает:

- Dockerfile для Web;
- Dockerfile для Worker;
- docker-compose с PostgreSQL;
- shared temporary storage volume;
- env examples без секретов;
- smoke-инструкции для офиса.

## Важные ограничения

- Админка не является клиентом бизнес-обработки PDF.
- Health/checks не должны запускать обычное распознавание пользовательского PDF.
- Возможный `InvalidInputProbe` через нулевой файл допустим только после отдельного безопасного contract checkpoint.
- Raw payload и входные PDF не показывать в Admin UI.
- Секреты не писать в audit, docs, support report и Git.
- Не использовать EF.
- DB-backed integration tests лучше гонять последовательно, чтобы не ловить гонки на общей PostgreSQL БД.
