# Карта архитектурных решений

Эта страница фиксирует не только решения, но и причины, почему выбран именно такой вариант.

## Модульный монолит вместо микросервисов

Решение:

```text
Модульный монолит по коду
```

Причина:

- меньше инфраструктурной сложности;
- проще разработка;
- проще тестирование;
- проще поставка;
- будущие модули можно добавлять без сетевого разбиения.

Что оставляем на будущее:

- возможность выделять отдельные сервисы, если появится реальная необходимость.

## Web и Worker как разные процессы

Решение:

```text
CenteralES.Web
CenteralES.Worker
```

Причина:

- обработка файлов долгая;
- API должен отвечать быстро;
- worker можно масштабировать отдельно;
- worker можно перезапускать без остановки админки;
- Web не должен зависеть от зависаний внешних обработчиков.

## Capability вместо прямого вызова сервиса

Решение:

```text
клиент вызывает capability
```

Причина:

- клиент не знает, ML это или не ML;
- можно менять реализацию;
- разные поставки могут использовать разные обработчики;
- UI админки становится понятнее.

## Processor Registry

Решение:

```text
создать registry возможностей и обработчиков
```

Причина:

- единое место для информации о доступных обработчиках;
- можно включать/выключать runtime;
- можно хранить endpoint, timeout, retry, health;
- можно строить Admin UI на основе этой модели.

## Асинхронная обработка

Решение:

```text
POST создаёт или возвращает job
GET получает status/result
```

Причина:

- PDF может быть больше 100 МБ;
- обработка может длиться больше 5 минут;
- синхронный HTTP плохо подходит для такого сценария.

## Дедупликация по hash

Решение:

```text
capability + content_hash
```

Причина:

- одинаковый файл может прийти из разных мест;
- повторная обработка тратит ресурсы;
- клиентам нужен общий jobId для отслеживания.

## PostgreSQL-backed queue

Решение:

```text
очередь задач в PostgreSQL
```

Причина:

- PostgreSQL уже нужен;
- меньше компонентов поставки;
- достаточно для MVP;
- можно использовать `FOR UPDATE SKIP LOCKED`.

Что оставляем на будущее:

- RabbitMQ, Redis, NATS или другой broker через `IJobQueue`.

## Локальный TemporaryFileStorage

Решение:

```text
локальное временное хранилище для входных файлов
```

Причина:

- входные файлы не хранятся для истории;
- проще первый релиз;
- не нужен S3/MinIO сразу.

Что важно:

- Web и Worker должны иметь доступ к файлам;
- нужна cleanup-политика;
- должна быть абстракция для будущего S3.

## Result index без большого payload

Решение:

```text
общий индекс результата + payload в таблицах подсистем
```

Причина:

- общая таблица не разрастается тяжёлыми JSON;
- у каждой подсистемы свои индексы;
- легче добавлять новые модули;
- меньше риск деградации производительности.

## API key A+

Решение:

```text
Authorization: ApiKey <keyId>.<secret>
```

Причина:

- просто для desktop-клиента;
- не тянем OAuth/JWT в MVP;
- можно сделать безопасно: hash secret-а, `keyId`, отключение, ротация, allowed capabilities.

## Админка для прикладного администратора

Решение:

```text
Admin UI в MVP, ориентированный на прикладного администратора
```

Причина:

- продукт нужно эксплуатировать;
- обработчики могут ломаться;
- нужны retry, включение/выключение, ключи, очередь;
- UI должен объяснять ситуацию понятным языком.

## Admin API логически, но не физически отдельно

Решение:

```text
/admin/api внутри CenteralES.Web
```

Причина:

- границы чистые;
- лишний микросервис не нужен;
- проще auth и deployment;
- можно вынести позже, если появится причина.

## Health-check-и без бизнес-обработки

Решение:

```text
Web live/ready + Worker heartbeat/local healthcheck
```

Причина:

- Docker нужен простой технический сигнал состояния контейнера;
- админке нужна подробная, но безопасная диагностика;
- Worker не должен становиться внешним HTTP-сервисом;
- внешние processor-ы без собственного health endpoint нельзя проверять через обычную обработку файлов;
- для `pdf2txt` состояние может быть `unknown`, пока нет реального вызова или подтверждённого safe diagnostic probe.

Что оставляем на будущее:

- расширенные diagnostic probe для processor-ов;
- отдельные health endpoint-ы у внешних сервисов, если они появятся;
- метрики Prometheus/OpenTelemetry, если появится эксплуатационная необходимость.
