# Цели и контекст

## Бизнес-контекст

Есть desktop-приложение сервиса электронного подписания. Для него нужен серверный компонент, который будет выполнять вспомогательные функции, не подходящие для выполнения на клиенте:

- тяжёлая обработка файлов;
- вызовы ML-сервисов;
- вызовы лицензированных сервисов;
- хранение результатов по hash;
- контроль обработки;
- администрирование поставки;
- будущие серверные возможности.

Система должна развиваться. Первый релиз не должен закрывать путь к новым подсистемам.

## Основная цель

Создать расширяемую серверную платформу для набора серверных служб сервиса ЭП.

Платформа должна позволять:

- подключать заранее описанные разработчиком обработчики;
- включать и отключать их в конкретной поставке;
- управлять runtime-настройками;
- принимать задачи от сервиса ЭП;
- обрабатывать задачи асинхронно;
- избегать повторной обработки одинаковых файлов;
- возвращать результаты по hash/job;
- показывать состояние системы в качественной админке.

## Что важно в первом релизе

Первый релиз должен быть не прототипом, а рабочей основой для расширения.

Критично:

- правильные архитектурные границы;
- понятная модель `Capability -> Processor -> Job -> Result`;
- качественный Admin UI;
- устойчивость к долгой обработке;
- контроль повторных попыток;
- понятная эксплуатация;
- минимальная инфраструктурная сложность.

## Что не делаем в первом релизе

В первом релизе не делаем:

- полноценную плагинную систему, где администратор сам создаёт новые обработчики через UI;
- тяжёлую микросервисную архитектуру;
- внешний брокер очередей;
- S3/MinIO как обязательную зависимость;
- OAuth/JWT-flow для desktop-клиента ЭП;
- сложные роли в админке;
- публичный внешний endpoint для админки.

## Почему не микросервисы сразу

Система должна быть расширяемой, но старт с тяжёлых микросервисов создаст лишние проблемы:

- больше deployment-компонентов;
- больше сетевых контрактов;
- сложнее отладка;
- сложнее транзакционные сценарии;
- больше инфраструктуры для первого релиза.

Лучше начать с модульного монолита по коду и нескольких процессов по эксплуатации.

## Почему worker отдельно от web

Обработка PDF и других файлов может занимать больше пяти минут. Файлы могут быть больше 100 МБ.

Если выполнять это внутри web-процесса:

- API может деградировать из-за тяжёлых задач;
- сложнее масштабировать обработку;
- сложнее перезапускать обработчики;
- выше риск зависания пользовательских запросов.

Поэтому Web и Worker должны быть разными процессами.

## Главный принцип

Desktop-клиент сервиса ЭП должен обращаться к понятным возможностям сервера, а не знать техническую реализацию:

```text
клиент просит: распознать штамп PDF
сервер решает: каким обработчиком это выполнить
```

Это позволяет менять реализацию без изменения клиентского API.
