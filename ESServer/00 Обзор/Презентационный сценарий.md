# Презентационный сценарий

Эта страница помогает рассказывать проект последовательно: от бизнес-смысла к архитектуре и MVP.

## 1. С какой проблемы начинаем

У desktop-приложения сервиса ЭП появляются функции, которые неудобно выполнять на клиенте:

- большие файлы;
- долгая обработка;
- ML-сервисы;
- лицензированные сервисы;
- централизованное хранение результатов;
- единый контроль ошибок и повторных попыток.

Поэтому нужен отдельный серверный компонент.

## 2. Почему это не просто PDF-сервис

Первый сценарий действительно связан с PDF:

- отправить файл;
- распознать штамп;
- найти авторов, должности и подписантов;
- вернуть JSON.

Но сразу известно, что дальше появятся другие подсистемы:

- обработка изображений;
- обработчики без ML;
- лицензированные продукты;
- будущие серверные функции.

Поэтому проектируется не один endpoint, а платформа серверных служб.

## 3. Главная идея платформы

Клиент ЭП должен просить не конкретный ML-сервис, а бизнес-возможность:

```text
Распознай штамп PDF
```

Сервер сам решает:

- какой обработчик включён;
- куда отправить файл;
- сколько раз повторять при ошибке;
- где хранить результат;
- что показать администратору.

## 4. Почему обработка асинхронная

PDF может быть больше 100 МБ.

Обработка может длиться больше 5 минут.

Поэтому нельзя делать основной сценарий как долгий синхронный HTTP-запрос.

Правильный flow:

```text
клиент отправил файл
  -> сервер принял в работу
  -> вернул hash и jobId
  -> worker обработал
  -> клиент получил результат по hash/job
```

## 5. Почему нужна дедупликация

Один и тот же файл могут отправить из двух мест.

Система должна:

- посчитать hash;
- понять, что файл уже обрабатывается;
- не запускать повторную обработку;
- вернуть тот же текущий jobId;
- дать всем клиентам получить один общий результат.

## 6. Почему нужен Worker

Web API должен отвечать быстро.

Тяжёлая обработка должна выполняться фоновой службой.

Поэтому:

```text
Web
  принимает запросы, отдаёт статусы, показывает админку

Worker
  выполняет долгие задачи
```

Это не микросервисная сложность, а нормальное разделение процессов внутри одного продукта.

## 7. Почему нужна админка в MVP

Система будет работать с внешними и лицензированными обработчиками.

Нужен интерфейс, где прикладной администратор может:

- увидеть, работает ли система;
- включить или выключить обработчик;
- проверить endpoint;
- увидеть очередь;
- увидеть failed jobs;
- запустить retry;
- управлять API keys.

Админка — часть продукта, а не второстепенный инструмент.

## 8. Как объяснить Processor Registry

Простая формулировка:

> Это каталог серверных возможностей и способов их выполнения.

Например:

```text
Возможность:
  распознать штамп PDF

Обработчик:
  внешний сервис pdf2txt
```

В другой поставке обработчик может быть другим, а возможность для клиента останется той же.

## 9. Как объяснить хранение результатов

Входные файлы временные.

Результаты хранятся как cache.

Общая таблица хранит только индекс результата, а большой payload лежит в таблице конкретной подсистемы или файловом storage.

Это нужно, чтобы система не замедлилась при добавлении новых модулей.

## 10. Что входит в MVP

MVP должен включать:

- Public API для сервиса ЭП;
- API key доступ;
- PDF stamp recognition;
- PostgreSQL-backed queue;
- Worker;
- TemporaryFileStorage;
- Result index и таблицу PDF payload;
- retry и дедупликацию;
- Admin UI;
- audit базовых действий;
- управление обработчиками и API keys.

## 11. Что остаётся на будущее

На будущее:

- полноценная плагинная модель;
- S3/MinIO storage;
- внешняя очередь;
- несколько ролей админки;
- HMAC/JWT;
- retention policy результатов;
- несколько активных processor-ов на одну capability;
- резервные обработчики.
